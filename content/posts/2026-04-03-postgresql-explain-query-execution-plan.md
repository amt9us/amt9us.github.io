PostgreSQL에서는 `EXPLAIN` 명령어를 통해 **데이터베이스가 쿼리를 실제로 어떤 방식으로 실행하는지** 확인할 수 있습니다.

처음에는 단순히 **쿼리 결과만 정상적으로 나오면 충분하다**고 생각했습니다.

> 굳이 실행 계획까지 확인할 필요가 있을까?

하지만 실제 개발 과정에서 이 생각이 바뀌게 되었습니다.

구간별 성능 테스트를 진행하던 중, 특정 구간에서 **유독 딜레이가 길게 발생하는 문제**가 있었습니다.  
더 이상했던 점은 **로컬 환경에서는 문제가 없었지만 개발 서버에서는 응답 속도가 눈에 띄게 느려졌다는 것**이었습니다.

처음에는 흔히 떠올릴 수 있는 방법들을 시도했습니다.

- 쿼리 구조 수정
- 인덱스 추가
- 조건문 변경

하지만 기대했던 만큼 성능이 개선되지는 않았습니다.

그때 **실행 계획(EXPLAIN)을 확인해보라는 조언을 듣게 되었고**,  
쿼리가 실제로 데이터베이스 내부에서 **어떤 방식으로 실행되고 있는지** 확인해보게 되었습니다.

그리고 그 과정에서 **예상과 다른 실행 방식이 선택되고 있다는 사실**을 발견할 수 있었습니다.

이 경험 이후로 DB 성능 문제를 분석할 때는  
**가장 먼저 실행 계획(EXPLAIN)을 확인하는 습관**을 가지게 되었습니다.

실제로 많은 경우, **DB 성능 문제의 원인은 실행 계획을 통해 빠르게 파악할 수 있기 때문입니다.**

## 1. 실행 계획(Execution Plan)이란 무엇인가

SQL을 실행하면 데이터베이스는 쿼리를 바로 실행하지 않습니다.

먼저 **Query Planner(쿼리 플래너)** 가 쿼리를 분석하여 가장 효율적인 실행 방법을 계산합니다.

전체 과정은 다음과 같습니다.

```text
SQL Query
   ↓
Query Parser
   ↓
Query Planner
   ↓
Execution Plan 생성
   ↓
Query Executor
```

작성한 SQL은 그대로 실행되는 것이 아니라, DB가 가장 효율적이라고 판단한 실행 전략으로 변환되어 실행됩니다.

이 실행 전략이 바로 **Execution Plan(실행 계획)** 입니다.

## 2. EXPLAIN을 확인해야 하는 이유

### 1. 인덱스 사용 여부를 확인할 수 있다.

예를 들어 `user_log` 테이블에 **1천만 건(10,000,000 rows)** 의 데이터가 있다고 가정해보겠습니다.

```sql
CREATE INDEX idx_user_id ON user_log(user_id);
```

`user_id` 컬럼에 인덱스를 생성했습니다.

이 상태에서 다음과 같은 쿼리를 실행합니다.

```sql
SELECT *
FROM user_log
WHERE user_id = '1001';
```

이 쿼리가 당연히 인덱스를 사용할 것이라고 생각했습니다.

하지만 실제 실행 계획을 확인해보면 결과가 다를 수 있습니다.

```sql
EXPLAIN
SELECT *
FROM user_log
WHERE user_id = '1001';
```

실행 결과 예시

```
Seq Scan on user_log
```

이 결과는 다음을 의미합니다.

> 인덱스를 사용하지 않고 테이블 전체를 순차적으로 스캔하고 있다.

이처럼 실제 실행 방식은 개발자의 예상과 다를 수 있기 때문에 실행 계획을 확인하는 과정이 중요합니다.

## 3. EXPLAIN vs EXPLAIN ANALYZE

PostgreSQL에서 실행 계획을 확인할 때는 두 가지 명령어를 사용할 수 있습니다.

### EXPLAIN

```sql
EXPLAIN
SELECT * FROM user_log WHERE user_id = '1001';
```

확인할 수 있는 정보

- 실행 계획
- 예상 cost
- 예상 row 수

하지만 실제 쿼리는 실행되지 않습니다.

즉, DB가 예상한 실행 계획만 확인할 수 있습니다.

### EXPLAIN ANALYZE

```sql
EXPLAIN ANALYZE
SELECT * FROM user_log WHERE user_id = '1001';
```

확인할 수 있는 정보

- 실제 실행 시간
- 실제 row 수
- 실제 실행 계획

즉 쿼리를 실제로 실행하면서 성능을 측정합니다.

그래서 쿼리 튜닝을 할 때는 보통 다음 순서를 사용합니다.

1. EXPLAIN 으로 실행 계획 확인
2. EXPLAIN ANALYZE 로 실제 성능 측정

## 4. 실행 계획을 보는 습관

DB 성능을 다룰 때 다음 순서로 확인합니다.

1. 쿼리 작성

```sql
SELECT ...
```

2. 실행 계획 확인

```sql
EXPLAIN
```

3. 실제 성능 확인

```sql
EXPLAIN ANALYZE
```
4. 인덱스 설계

```sql
CREATE INDEX
```

5. 실행 계획 비교

```text
Before / After 비교
```

이 과정을 통해 쿼리 성능이 실제로 개선되었는지 확인할 수 있습니다.

---

SQL은 단순히 **동작하는 것**보다 **어떤 방식으로 실행되는지**가 더 중요하다는 것을 이번 경험을 통해 알게 되었습니다.

특히 데이터가 많은 서비스에서는 작은 차이 하나가 큰 성능 차이를 만들 수 있습니다.  
개발 환경에서는 문제가 없던 쿼리도 데이터가 많아지는 순간 예상하지 못한 병목이 발생할 수 있습니다.

이러한 문제를 분석할 때 **실행 계획(Execution Plan)** 은 매우 중요한 도구입니다.  
실행 계획을 확인하면 쿼리가 실제로 어떤 방식으로 데이터를 조회하고 처리하는지 파악할 수 있습니다.

앞으로는 쿼리를 작성할 때 결과만 확인하는 것이 아니라  
**실행 계획을 함께 확인하는 습관**을 가져야겠다고 느꼈습니다.
